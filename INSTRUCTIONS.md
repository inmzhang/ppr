# PPR — Ping-Pong Review

Portable instructions for any AI coding agent participating in a PPR review.
These instructions work with Claude Code, Codex, Gemini CLI, OpenCode, or any
terminal-based coding agent.

## What is PPR?

PPR is a multi-round code review protocol where AI coding agents review each
other's work. One agent (the **author**) initiates a review after implementing
changes, and other agents (the **reviewers**) provide structured feedback. The
author addresses the feedback and responds, and reviewers re-review — back and
forth like a ping-pong rally — until everyone agrees the code is good.

## The `ppr` CLI

All protocol operations go through the `ppr` shell script. The key commands:

| Command       | Purpose                                           |
|---------------|---------------------------------------------------|
| `ppr init`    | Start a new review session                        |
| `ppr launch`  | Launch reviewer agents in the background           |
| `ppr status`  | Check review state and detect completed reviews    |
| `ppr wait`    | Block until all reviews for this round arrive      |
| `ppr collect` | Print all reviews for the current round            |
| `ppr respond` | Address feedback and advance to the next round     |
| `ppr finish`  | Generate a summary and optionally commit           |
| `ppr clean`   | Remove the .ppr/ directory                         |

## Protocol: File-Based Communication

Agents communicate through files in a `.ppr/` directory at the project root.
This is the most portable approach — every terminal agent can read and write
files.

### Directory Layout

```
.ppr/
├── config.json              # Review configuration (reviewers, max rounds)
├── status.json              # Live state (current round, reviewer status)
├── agents.json              # (Optional) project-local agent launch configs
├── round-1/
│   ├── changes.diff         # The diff under review
│   ├── context.md           # Author's description of the changes
│   ├── reviews/
│   │   ├── reviewer-1.md    # Each reviewer writes their feedback here
│   │   └── reviewer-2.md
│   └── response.md          # Author's response to the reviews
├── round-2/
│   └── ...                  # Same structure, new diff after author's changes
└── summary.md               # Generated by 'ppr finish'
```

### Status Tracking

`status.json` tracks the live state of the review:

```json
{
  "current_round": 1,
  "max_rounds": 3,
  "reviewers": {
    "reviewer-1": { "status": "pending", "lgtm": false },
    "reviewer-2": { "status": "reviewing", "lgtm": false }
  },
  "state": "awaiting_reviews"
}
```

Reviewer status progresses: `pending` → `reviewing` → `done`.
The overall state is one of: `awaiting_reviews`, `consensus`, `max_rounds_reached`.

## Workflow: As the Author (Initiating Agent)

You are the author when you've just finished implementing a feature and want
other agents to review your changes.

### 1. Initialize the review

```bash
ppr init \
  --reviewers "claude-code:alice,codex:bob" \
  --max-rounds 3 \
  --context "Added retry logic to the HTTP client"
```

This creates `.ppr/` with your current git diff and the review configuration.
The `--reviewers` flag takes a comma-separated list of `agent-type:name` pairs.

### 2. Launch the reviewers

```bash
ppr launch
```

This starts all reviewer agents in the background. Each one receives a prompt
with your diff, context, and instructions on where to write their review.

### 3. Wait for reviews

```bash
ppr wait --timeout 600
```

This blocks until all review files appear (or times out). You can also poll
with `ppr status` if you prefer non-blocking checks.

### 4. Read the reviews

```bash
ppr collect
```

This prints all reviewer feedback for the current round.

### 5. Address the feedback

Read each review and decide how to respond:

- **Agree**: Make the code change, then explain what you changed.
- **Disagree**: Write a clear argument for why the current approach is better.
- **Question**: Ask for clarification.

After making your changes:

```bash
ppr respond --message "Fixed the race condition bob flagged. Disagree with
alice on the naming — 'retry_count' is clearer than 'attempt_number' because
it excludes the initial request."
```

This captures your new diff and advances to the next round.

### 6. Repeat until consensus

Run `ppr launch` again for the new round. Reviewers who already gave LGTM
are skipped — only those who requested changes are re-launched.

The review ends when:
- All reviewers signal **LGTM**, or
- The max round limit is reached

### 7. Finish and commit

```bash
ppr finish --commit --message "feat: add HTTP client retry logic"
```

This generates `summary.md` with the full review history and creates a commit.

## Workflow: As a Reviewer

You are a reviewer when you've been launched by another agent to provide
feedback on their changes.

### What you receive

When launched, your prompt contains:
- The **diff** of the changes under review
- The author's **context** explaining what they changed and why
- For round 2+: your **previous review** and the author's **response**
- The **review output path** for your final answer

### How to write your review

1. Read the diff carefully. Consider: correctness, security, performance,
   readability, and project conventions.

2. For each issue you find, include:
   - The **file and line** reference
   - A **category tag**: `[bug]`, `[security]`, `[performance]`, `[style]`,
     `[suggestion]`, `[question]`
   - A clear **explanation** of the problem
   - A **suggested fix** when possible

3. End your review with a verdict:
   - **LGTM** — no blocking issues (minor suggestions are OK alongside this)
   - **CHANGES REQUESTED** — issues that should be addressed before merging

4. **Emit only your final markdown review.** If your CLI supports a dedicated
   "final output to file" flag, use the provided review path for that final
   answer only. Otherwise print the final review to stdout. Do not edit files
   or apply patches.

### Example review

```markdown
## Review of HTTP Client Retry Logic

Overall this looks solid. The exponential backoff approach is appropriate.
Two issues:

### [bug] src/http_client.rs:47
The retry loop doesn't check for non-retryable status codes (e.g., 400, 401).
A malformed request would retry 3 times before failing.

Suggested fix: add a check for `status.is_client_error()` before retrying.

### [suggestion] src/http_client.rs:52
Consider adding jitter to the backoff delay to avoid thundering herd when
multiple clients retry simultaneously.

**CHANGES REQUESTED**
```

### Reviewing in subsequent rounds

In round 2+, you'll see the author's response to your previous feedback and
a new diff showing what changed. Focus on:
- Whether the author addressed your concerns
- Whether the new changes introduce new issues
- The author's arguments for any disagreements — engage thoughtfully

If all your concerns are addressed, write **LGTM**.

## Agent Configuration

The `ppr` script needs to know how to launch each type of agent. It reads
launch commands from `agents.json`, checking:

1. `.ppr/agents.json` (project-local, takes priority)
2. `~/.config/ppr/agents.json` (user-global default)

### Format

```json
{
  "claude-code": {
    "cmd": "claude -p --permission-mode plan --output-format text \"$(cat \"$PROMPT\")\""
  },
  "codex": {
    "cmd": "codex --ask-for-approval never exec --sandbox read-only -o \"$REVIEW_FILE\" - < \"$PROMPT\""
  },
  "gemini": {
    "cmd": "gemini -p \"$(cat \"$PROMPT\")\" --approval-mode plan --output-format text"
  },
  "opencode": {
    "cmd": "opencode -p \"$(cat $PROMPT)\""
  }
}
```

The `$PROMPT` placeholder is replaced with the path to a file containing the
reviewer's full prompt (diff, context, instructions). The optional
`$REVIEW_FILE` placeholder is replaced with the final review path. The command
should:

- Run **non-interactively** (print/pipe mode)
- Run in **read-only / no-write review mode** when the agent supports it
- Either print the final answer to **stdout** or write only the final answer
  to **`$REVIEW_FILE`**

### Adding a new agent type

To add support for any terminal-based coding agent:

1. Figure out its non-interactive invocation and the safest review-only permission mode
2. Add an entry to `agents.json` with the command template
3. Use `$PROMPT` where the prompt file path should go
4. Use `$REVIEW_FILE` if the CLI can write just its final answer directly

## Tips

- **Start small**: Try a 2-reviewer setup before scaling up.
- **Be specific in context**: The better your `--context` message, the more
  focused the reviews will be.
- **Engage with disagreements**: When a reviewer pushes back on your response,
  consider their perspective seriously. The goal is better code, not winning.
- **Watch the round limit**: 3 rounds is usually enough. If a review is still
  contentious after 3 rounds, the issue might need a design discussion rather
  than code review.
